
### 统一使用 “###（H3 标题）+ 🧠 + [知识点名]” 的方式作为知识点

- 整篇文章本质上就是将各个知识点按照最优的叙事逻辑与合理的递进顺序进行精心编排。

- 🧠 符号表达：需要记忆/思考的内容，打破纯文字的沉闷，给每段知识一个“锚点”。

- H3 (###) 标题在大纲视图（Outline）中，你可以一眼看到所有的知识点，无需在正文中慢慢往下翻找，非常利于快速跳转。

- 为什么是 H3 ？# 常用做文件名 title，## 用于给知识点分类，所以只能用 H3 了，为什么不用 H4？因为 H4 标题在 VSCode 的 preview 中看起来很小

- 文章结构只应有两层：#（文件的大标题）和 ###（知识点），不要使用 ##（我会自己手动给知识点分类）。

- **示例：**

    - ```markdown
        ### 🧠 知识点的名称

        - **该解释段的关键词：** 知识点的解释 A。

        - **该解释段的关键词：** 知识点的解释 B。

        - **示例：** 知识点的示例。
      ```

### 关键词摘要索引

- **怎么做？**

    - 在每一个超过 30 个字符的段落的最前面，用 “**段落内容的高度概括关键词** 段落内容” (加粗的关键词，后跟一个空格，再跟段落内容) 的方式总结段落最核心的一句话，后面再跟大段文字，充当段落的索引。

- **为什么？** 我们要为大脑设计适合快速查阅的 “段落索引”，让读者视线扫过长文档时，在 0.5 秒内就能通过段首的黑体字决定是否细读，扫读（Skimming）极快。即便不看后面的细节，读第一句也能掌握 80% 的知识。


- **注意：**
    - **严禁行内强调**：不再将 `**加粗**` 散落在长句中部用于表达“强调语气”，避免视觉噪点切碎阅读心流。
    - 短段无需加索引。

- **示例：**

    - **✅ 正确示范：**

        - **大小写玄学** 在 Windows 系统中，文件名是不区分大小写的。

        - **末尾陷阱** 文件名结尾绝不能留有空格或小数点。

    - **❌ 错误示范：**

        - 在 Windows 系统中，文件 **绝对不区分** 大小写。

        - 文件名结尾绝不能留有 **空格** 或 **英文句点**。


### 知识点的每个解释段用无序列表

- **为什么？** 在 VSCode 的 Preview 里，H3 下的 `-` 句子会自动往右缩进，这样就有层级的观感，易于阅读。

- **怎么做？**

    1. 在每个 H3 下，用无序列表 `-` 来表示并列的解释段落。

    2. 每个 `-` 项之间用一个空行隔开。

    3. 如果 `-` 项还有子项，子项继续用缩进的 `-` 表示。

    4. 如果 `-` 项的子项是有顺序的序列，用缩进的有序列表（1. 2. 3.）表示。

    5. 如果 `-` 项的子项是代码块，用缩进的代码块表示。`-` 符号后面加一个空格，然后跟代码块的开头 ```，然后是代码块的语言，再跟一个换行符，然后是代码块的内容，再跟一个换行符，然后是代码块的结尾 ```。结尾的 ``` 必须和开头的 ``` 垂直于同一列对齐。注意代码块的格式很重要，错误的格式会直接导致渲染失败，请严格按照下面的模板输出代码块。

- **示例**

    - 这是一个短的解释段落。

        - 这是子段

    - **该长解释段的关键词：** 知识点的解释长段落............................................................................................

    - 这个短段的子段是有序列表

        1. 子段 A

        2. 子段 B

    - ```go
      func main() {
          fmt.Println("Hello, World!")
      }
      ```
