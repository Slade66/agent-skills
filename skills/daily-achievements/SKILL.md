# Git 每日成果整理

## 工作流程

1. 除非用户明确要求修改文件，否则把所有仓库都视为只读对象。

2. 确定日期范围和时区。优先使用 `git log` 显示的提交时间；如果用户给出的是本地日期，就使用本地自然日边界，例如 `YYYY-MM-DD 00:00:00` 到 `YYYY-MM-DD 23:59:59`。

3. 使用 `git log --all` 检查每个仓库，信息应包含 commit hash、提交时间、提交标题和改动文件。

4. 先按自然日分组，再在每一天下面按仓库分组。

5. 每一天都要列出用户要求的所有仓库。如果某个仓库当天没有提交，写 `无提交`。

6. 把 commit 转换为成果表述，但每条成果都必须能追溯到具体标题、文件、API、模块、脚本或文档。

## 命令

使用类似下面的命令，并根据用户请求调整路径和日期范围：

```powershell
git -C "A:\path\repo" log --all `
  --since="2026-04-19 00:00:00" `
  --until="2026-04-28 23:59:59" `
  --date=iso-strict `
  --pretty=format:"@@@%h`t%cd`t%s" `
  --name-status
```

如果 commit message 很模糊，例如 `1`、`提交代码`、`修改代码` 或 `update`，继续检查改动文件。

对于 Markdown 笔记，优先参考 commit subject 和文档 H1/title。如果二者不同，使用更具体、更适合人阅读的标题。

## 精确性规则

- 不要单独使用 `一篇笔记`、`相关功能`、`一些图片`、`做了整理`、`补充资料`、`职场事件复盘` 这类模糊表述。

- 对笔记仓库，写出精确笔记标题，例如 `新增《XXX 的反思》`。

- 对代码仓库，写出精确交付物，例如 `新增 ESXi 主机 CRUD`、`实现 bifrost-migrate CLI 骨架`、`新增 FusionCompute datastore/portgroup 查询接口` 或 `新增《Bifrost WebSocket 实时推送协议 v1》`。

- 对文档提交，写出精确文档标题。

## 输出格式

默认使用下面的结构：

```markdown
**YYYY-MM-DD**
- `repo-one`：具体成果。
- `repo-two`：无提交。
- `repo-three`：具体成果。
```

## 质量检查

最终回复前检查：

1. 确认请求范围内的每个日期都出现了。
2. 确认每个日期下面都列出了用户要求的所有仓库。
3. 把模糊摘要替换为具体标题、模块、接口或文件。
4. 确保无提交的仓库明确写了 `无提交`。
5. 保持措辞适合直接放入每日成果记录。
